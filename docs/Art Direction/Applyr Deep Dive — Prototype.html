The Mission
Applyr eliminates the 80% of job search effort wasted on poor-fit roles and generic documents. A user with a configured profile goes from "scout triggered" to "PDF-ready tailored application" with zero manual effort per job.

What It Actually Does

The core user flow is deceptively simple to describe and genuinely difficult to build: configure once, then let the machine find, score, and draft — repeatedly, for every role, without hallucinating credentials.

The Six-Step Flow

  1. Configure — target role, blocklists, salary floor, and LLM API keys via settings UI
  2. Codify — paste full work history; system auto-assigns stable proof codes (ACC-101, VOC-01, MET-03)
  3. Scout — headless browser + 7 API sources scrape new postings matching criteria
  4. Score — zero-token keyword gate first; LLM scoring only for jobs that pass
  5. Draft — qualifying roles get company research, tailored resume, cover letter, and interview cheat sheet compiled to PDF
  6. Track — WYSIWYG editor for final edits; 6-stage lifecycle from New to Closed

The Problem It Solves

Job searching at scale requires applying to many roles with tailored documents. Both parts are painful: sourcing (finding relevant roles across many sites) and tailoring (customizing resumes without hallucinating credentials). Together they make high-volume quality search impractical by hand.

"Commercial AI resume tools are subscription SaaS, send your career data to their servers, and don't integrate scouting with drafting into one pipeline."

— Project Deep Dive, Part 1.2

Target User

A mid-career technical individual contributor actively job searching who: (a) has a strong but nuanced work history that doesn't translate well to generic templates, (b) is comfortable running a local dev environment, (c) applies to 10–30 roles per month and wants each one properly tailored.

Origin Story

Started as a LinkedIn scraper — simple Playwright automation to collect job posts matching a specific role. Scope expanded significantly through versioned change requests. The scouting layer grew from 1 source to 8. Scoring was added when raw scraping produced too much noise. Drafting came when scoring identified good roles but manual tailoring remained the bottleneck. The UI arrived when managing everything via CLI became unwieldy.

Key Pivots

  • CR-001 — Playwright-only scraping → hybrid pipeline (APIs + Playwright)
  • CR-003 — Four disconnected preference UIs → one materialized JSON pipeline (ADR-005)
  • CR-005 — .env key management → SQLite + UI (security concern: accidental git commits)
  • CR-006 — Single LLM provider → fallback chain after recognizing silent failures from outages

Architecture
& Stack
Five major components across two languages, three runtimes, and a file system used as a message queue. The architecture was shaped by a privacy constraint, a single-user assumption, and the fact that all the LLM code already existed in Python.

System Architecture

The architecture is a hub-and-spoke model: Express at the center, spawning child processes for all heavy work. The frontend polls for state; the backend never pushes proactively.

Fig 1 — System architecture: Express hub, SQLite state, four spawned child processes
Key Design Insight

The server never awaits Python processes — they run as detached child processes writing progress to stdout. The frontend polls GET /api/logs every 5 seconds, accepting up to 5 seconds of lag in exchange for zero WebSocket complexity.

Technology Stack

Every technology choice was shaped by the same two constraints: privacy (local-only, no cloud) and velocity (the developer is also the user). The trade-offs are documented honestly.

Chapter 03
The Decision
Ledger
Seven architectural decisions — each documented with the context that forced the choice, the options that were considered, and the trade-offs accepted. Honest about what was vibe-coded and what was deliberate.

Seven Architectural Decisions

The best architectural decisions are made under constraint. Every decision below was shaped by privacy, cost, skill, or velocity. None were made in a vacuum.

Decision #1
Local-First Architecture
Why This Won

Privacy constraint was non-negotiable — the app stores a user's full work history, target companies, and job search strategy. Cloud would mean career intelligence on third-party servers. The developer's skill set aligned with web stack. Single user means zero multi-tenancy complexity. The target user can clone a repo and run npm run dev.

Options considered: Cloud SaaS (Vercel + Supabase) — rejected for privacy and recurring cost. Electron app — rejected for heavy bundle and platform-specific build complexity.

Trade-off accepted: Non-technical users cannot use this. No mobile. No sharing. No multi-machine access. Revisit if project ever targets non-technical users.
Decision #2
SQLite Over PostgreSQL
Why This Won

better-sqlite3's synchronous API was the tipping point. Express route handlers that do db.prepare(...).run() are cleaner than async await pool.query(...) chains. The single-user constraint meant write contention was never a concern.

Options considered: PostgreSQL (requires running service, overkill for 1 user), JSON files (no querying, no atomic writes, race conditions with concurrent pipeline processes).

Trade-off accepted: Single writer, file-level locking. Revisit only if multiple users, write throughput > ~100/second, or data volume > ~50GB.
Decision #3
Python Pipeline as Separate Process
Why This Won

All the LLM and PDF logic already existed in Python. Rewriting in Node would have been a net regression. The spawn pattern gave "good enough" IPC for a single-user app — stdout capture provides real-time log streaming, no shared memory concerns.

Options considered: Inline Node.js LLM calls (would require rewriting all Python scripts), FastAPI microservice (port management complexity, startup ordering).

Trade-off accepted: Stdout parsing is fragile (relies on [LOG], [FOUND] prefix conventions). No mid-run cancellation. Process exit code is the only reliable error signal.
Decision #4
Materialized JSON for Pipeline Config (ADR-005)
Why This Won

Clean separation of concerns. The JSON file is a cache/projection of SQLite state. The server owns the write path; Python owns the read path. No cross-language DB connection needed. Decoupled (Python doesn't know about SQLite), human-readable, and diffable for debugging.

Options considered: CLI args (shell escaping nightmares on Windows with 10+ settings), Python reads SQLite directly (two concurrent connections, Python needs DB schema knowledge).

Trade-off accepted: Two sources of truth. Drift risk if server crashes mid-write. In practice: materialization is synchronous and immediately follows the SQLite write. Verdict: this pattern is correct for this architecture.
Decision #5
LLM Provider Fallback Chain (CR-006)
Why This Won

The pipeline runs unattended, often during hours when the user isn't watching. Silent recovery is more valuable than explicit failure in an unattended context. A rate limit at 2am should not kill an entire overnight scout run.

Options considered: Single provider, hard fail (clear signal but kills unattended runs), user manually switches provider (requires user to notice the failure and intervene).

Trade-off accepted: A job might be scored by Gemini and a second by Claude in the same batch, producing inconsistent scoring. Logged but easy to miss. Revisit if audit trails of which provider scored which job become important.
Decision #6
Proof Code Anti-Hallucination System
Why This Won

The structural constraint is more reliable than a prompt constraint. LLMs reliably ignore "only use facts from this document" under token pressure. The proof code system structurally prevents hallucination — if no code exists, the claim cannot be made. This was the founding design decision that differentiates Applyr from "just call ChatGPT with my resume."

Options considered: Post-generation verification (second LLM call, catching hallucinations after is harder than preventing them), strict prompt constraints (LLMs ignore this under token pressure).

Trade-off accepted: High setup cost. User must invest time codifying their experience before the pipeline produces useful output. No shortcut exists. The workExperience_summary.md auto-generation reduces ongoing maintenance but initial setup remains manual.
Decision #7
Two-Stage Scoring — Zero-Token Gate + LLM
Why This Won

Cost constraint. Gemini Flash free tier has limits. Burning them on obvious rejects is wasteful. Real-world results: ~70% of jobs are gated out before any LLM call. Token cost is roughly proportional to signal quality — only interesting jobs get the expensive treatment.

Options considered: LLM for everything (most accurate, ~50 calls per run), zero-token gate only (free and instant, but misses nuance like a growth-PM role titled "Product Manager").

Trade-off accepted: False negative risk at the gate layer. Mitigated by keeping the gate broad (keyword presence, not match score) and truncating JDs at 1500 chars rather than rejecting on length.
Chapter 03 — Key Points

The Decision Ledger

  • Local-first was non-negotiable — privacy constraint drove the entire architecture
  • SQLite won on API ergonomics (synchronous better-sqlite3) not just simplicity
  • Decisions #4 and #6 (materialized JSON, fallback chain) were explicitly designed — not vibe-coded
  • Decision #6 (proof codes) is the core IP of Applyr — structural anti-hallucination beats prompt engineering
  • Decision #7 (two-stage scoring) was cost-driven: spend tokens proportional to signal strength

The Pipeline
From scout trigger to PDF delivery. Four chained child processes, a two-stage scoring funnel, five LLM use cases, and a proof code system that makes hallucination structurally impossible.

Scout Pipeline — End to End

User clicks "Run Scout." Over the next 5–20 minutes, the system discovers new job postings from 7 sources, scores them, and generates application PDFs for qualifying roles. The user sees live log output in a terminal-style view.

Fig 2 — Two-stage scoring funnel: keyword gate eliminates ~70% of jobs before any LLM token is spent

Proof Code Anti-Hallucination System

The founding differentiator. Every claim in workExperience.md gets a stable, employer-partitioned code. Generated documents must cite these codes. The LLM is structurally forbidden from inventing uncoded claims.

Critical Failure Mode

If a user deletes a coded line instead of retiring it ([RETIRED-ACC-101]), future codification may re-assign that number to a different claim. This corrupts any existing document that cited the old code.

The Fallback Chain — Core Code

scripts/utils.py — Provider fallback chain
def _get_configured_providers(settings: dict) -> list:
    # primaryProvider with backward compat for old 'provider' field
    primary = settings.get('primaryProvider') or settings.get('provider', 'gemini')
    fixed_order = ['gemini', 'claude', 'local', 'perplexity']
    ordered = [primary] + [p for p in fixed_order if p != primary]
    return [p for p in ordered if _is_configured(p, settings)]

def call_llm(...):
    providers = _get_configured_providers(settings)
    for i, provider in enumerate(providers):
        result = None
        if provider == 'gemini':
            result = _call_gemini(...)   # Returns None on error, str on success
        elif provider == 'claude':
            result = _call_claude(...)   # tools param intentionally omitted
        if result is not None:
            return result                # str result terminates the loop
        if i + 1 < len(providers):
            print(f"    [LLM] Falling back from {provider} to {providers[i+1]}...")
    return ""  # All providers failed — callers must handle empty string
Known Degradation on Fallback

The tools=[{"google_search": {}}] parameter used by company research is only valid for Gemini. When falling back to Claude or Local, the tools param is dropped. Company research quality degrades on fallback — no web grounding means potentially stale competitive intel.

The Pipeline

  • Four sequentially chained child processes: Scout → Backfill → Scraper → Python Pipeline
  • 70% of jobs are eliminated by the keyword gate before any LLM token is spent
  • Total batch cost: <$0.015 per run at Gemini Flash rates — fits entirely within free tier
  • Proof codes make hallucination structurally impossible: no code = no claim
  • Fallback chain survives single-provider outages; trades consistency for resilience

Core Schema

The database is minimal by design. Single user, single writer, no joins beyond simple lookups. The interesting architectural decisions are in what was not normalized.

The jobs Table

server/db.ts — jobs table (primary entity)
id            TEXT PRIMARY KEY   -- UUID. randomUUID() in server/index.ts:259
company       TEXT NOT NULL      -- Denormalized. No companies table. Simpler queries.
title         TEXT NOT NULL      -- As scraped
url           TEXT UNIQUE        -- THE deduplication mechanism for the entire pipeline
score         INTEGER            -- 0-100 fit score; NULL before evaluation
status        TEXT DEFAULT 'Drafted'  -- 6-stage lifecycle
summary       TEXT               -- One-line LLM summary
salary_range  TEXT               -- Freeform. Avoids a parsing problem not worth solving.
source_site   TEXT               -- LinkedIn / Himalayas / RemoteOK / etc.
rejection_stage TEXT             -- At what stage was this closed
rejection_type  TEXT             -- Rejected / Ghosted / Withdrew
interview_date  DATETIME         -- Calendar awareness
created_at      DATETIME         -- Auto-set on insert
The URL UNIQUE Constraint
The entire scout pipeline's deduplication strategy fits in one SQL constraint. INSERT OR IGNORE on URL means the same posting scraped twice creates no duplicate. The UNIQUE constraint does the work that would otherwise require application-level dedup logic — elegant and zero-overhead.

The profiles Table — KV Store

server/db.ts — profiles table (all user configuration)
key   TEXT PRIMARY KEY  -- 'identity' | 'llm_settings' | 'job_search' | 'api_connections'
value TEXT              -- JSON-serialized blob. JSON.parse() at read time.

-- Adding a new settings category = zero schema migration, just a new key.
-- Downside: no SQL-level validation of JSON structure.

The activity_log Table — Append-Only Event Log

server/db.ts — activity_log (the "Terminal" view data source)
id        INTEGER PRIMARY KEY AUTOINCREMENT
timestamp DATETIME DEFAULT CURRENT_TIMESTAMP
level     TEXT NOT NULL   -- INFO | WARN | ERROR
source    TEXT NOT NULL   -- Scout | Pipeline | Scraper | System
message   TEXT NOT NULL
meta      TEXT            -- Optional JSON blob for structured data

-- Pull model, not push. Frontend polls GET /api/logs every 5 seconds.
-- Up to 5-second lag. Acceptable for a background process running over minutes.
-- TECHNICAL DEBT: table never pruned. Will degrade query performance over months.

Schema Evolution — Honest Assessment

Technical Debt — Critical

No migration system. The schema uses CREATE TABLE IF NOT EXISTS which is idempotent but doesn't handle column additions. New columns require manual ALTER TABLE or deleting the database entirely (losing all job history). This is acceptable for a solo-dev app; it's a hard blocker for any public distribution.

What Data Is Lost on Server Restart

  • Pipeline running status — resets to idle on restart. Pipeline process may still be running as a detached child; UI loses visibility but jobs still write to SQLite.
  • SSE stream connections — ephemeral, not persisted. Client must reconnect.
  • Draft SSE progress — if server restarts mid-pipeline, the user sees no log output, but new jobs will appear on the next polling cycle.

What Was Optimized

  • SCORING_JD_MAX_CHARS = 1500 — Truncating JDs reduces input tokens 60–70% with minimal signal loss (requirements are front-loaded in most JDs)
  • workExperience_summary.md for scoring — Full file only on YES decision. Rejected jobs never see the full 5,000-word experience doc.
  • Zero-token keyword gate — Eliminates LLM calls for obvious rejects. Calibrated to be broad, not precise.

What Was Intentionally Ignored

  • No caching of LLM responses — same company re-evaluated on every scout run
  • No parallel LLM calls — jobs processed sequentially in batch_pipeline.py
  • No Playwright browser reuse — each spawn starts a fresh Chromium instance

Security Surface

By Design — Acceptable

No authentication, no CORS restriction, no HTTPS, no rate limiting. All are intentional trade-offs for a localhost-only single-user app. An attacker with local machine access has far worse attack surfaces available than this API.

API Key Protection

  • Keys stored in SQLite profiles table — jobagent.sqlite is gitignored
  • Keys injected at spawn time as environment variables, never written to log files or API responses
  • .env excluded from git and documented as optional fallback only
Unmitigated Risk — Prompt Injection

Job descriptions scraped from the web are injected directly into LLM scoring prompts. A malicious JD could contain prompt injection text. Partially mitigated by structured JSON output enforcement and low temperature (0.1). Not fully mitigated — a sophisticated injection could manipulate scoring output.

Learnings Worth Keeping

"Structural constraints are more reliable than prompt constraints. If no code exists, the claim cannot be made — this is more reliable than 'only use facts from this document.'"

— Learning extracted from the proof code system

Patterns Worth Reusing

  • Materialized view pattern — Write to DB, project to a file that downstream consumers can read without DB knowledge. Clean separation, debuggable, diffable.
  • Two-stage evaluation — Zero-cost gate first, expensive gate second. Applicable to any pipeline with a large input set and expensive evaluation.
  • Proof code anti-hallucination — Structural constraints (cite a code or don't make the claim) beat prompt constraints universally.
  • call_llm() fallback chain — General-purpose pattern for any multi-provider LLM app. Return None to try next; return str (even empty) to stop.

Mistakes Not to Repeat

  • Building a server monolith — The route boundary was always clear; it just took 844 lines of pain to enforce it. Extract route handlers from day one.
  • Not designing IPC from the start — The [LOG]/[FOUND] stdout prefix convention was a hack that became load-bearing. Define a JSON streaming protocol at the start.
  • Not adding migrations early — Schema changes are painful. SQLite ALTER TABLE is limited. Add a migration table before writing the first schema.

Abstraction Drift — Honest Assessment

Thought vs. Reality Gap
"The settings UI manages all configuration" — Reality: data/job_fit_engine.md is a manually maintained prompt artifact that governs scoring. Changes to scoring criteria require editing this file directly. No UI exists for it. It's configuration, but it's really a prompt component that acts like code.

"The pipeline processes new jobs" — Reality: the jobs/ directory accumulates .txt files from every scout run. Old files are never cleaned. If a company changes its listing (same URL, updated JD), the old text file persists and the pipeline scores using stale text.

Craft & Debt

  • Performance is dominated by Playwright cold starts and LLM latency — both are network/compute, not code quality issues
  • The JD truncation at 1500 chars was empirically tuned: longer truncation improved recall <5% while increasing cost ~40%
  • Security surface is intentionally minimal for localhost-only use; prompt injection is the one unmitigated real risk

Roadmap

Three Phases to Production-Grade

The app works today. These phases represent what it would take to make it robust enough for distribution beyond a single technical user.

Phase 1 — Critical for Robustness
Foundation Fixes
  • Add ALTER TABLE migration system before any further DB schema changes
  • Clean up jobs/ directory after each pipeline run — delete processed files
  • Add LIMIT to activity_log queries; periodic DELETE for records older than 30 days
  • Replace stdout prefix parsing with newline-delimited JSON from Python scripts
Phase 2 — Important for Reliability
Pipeline Hardening
  • Introduce a lightweight job queue for pipeline stages — even a SQLite-backed queue beats callback pyramids
  • Add per-job LLM call timeout to prevent a single slow call from blocking the entire batch
  • Add structured logging of token counts and provider used per LLM call
  • Playwright browser pool/reuse across scout stages — eliminate cold start overhead
Phase 3 — Nice to Have
Developer Experience
  • UI for job_fit_engine.md editing — scoring rules are currently invisible to non-technical users
  • Unit tests for codification engine; integration tests for pipeline stages
  • Structured error propagation from Python — typed error objects instead of stderr strings
What to Keep on a Full Rebuild

SQLite for this use case — it's the right tool. The two-stage scoring architecture. The proof code anti-hallucination system. The materialized JSON projection pattern. The LLM fallback chain. These four patterns are the durable intellectual value of this codebase.

Start with a job queue. Define a JSON streaming protocol for Python → Node communication from day one. Extract route handlers into modules before the second route is written. Add a migration system before writing the first schema.
