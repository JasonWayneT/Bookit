The Core User Flow
1 Configure — Target role, blocklists, salary floor, and LLM API keys via a settings UI.
2 Import experience — Paste full work history; system auto-assigns ACC/VOC/MET proof codes to every claim.
3 Scout — Headless browser + 7 API sources scrape new postings matching the configured criteria.
4 Score — Each job is evaluated by an LLM against the user's profile. A zero-token keyword gate runs first to eliminate obvious rejects without spending any API tokens.
5 Draft — Roles above threshold get a company research packet, then a tailored resume, cover letter, and interview cheat sheet compiled to PDF.
6 Apply — Review and edit assets in a WYSIWYG editor, then track the role through six lifecycle stages to Applied or Closed.
System Architecture

Five components across three runtimes — a React frontend, an Express server, a TypeScript scout engine, a Python pipeline, and a SQLite database connecting them all. As of May 2026, the Express server is fully modularized: server/index.ts is 33 lines of middleware and router mounts, with all route logic split across server/routes/ and shared utilities in server/shared.ts.

System Architecture · Applyr v5.17 · server/index.ts now 33 lines after May 2026 router split
Why Not a SaaS Tool?

LinkedIn Easy Apply produces generic applications. Resume builders require manual per-role editing. Commercial AI tools like Teal and Rezi are subscription SaaS — they send your career data to their servers, and don't integrate scouting with drafting into one pipeline.

Applyr is local-first. All career data stays on disk. The only data sent externally is what you explicitly configure: JD text to LLMs for scoring, company names to research APIs. No account, no subscription, no shared infrastructure.

Primary Value

A user with a configured profile can go from "scout triggered" to "PDF-ready tailored application" with zero manual effort per job. The pipeline handles sourcing, filtering, and drafting — the user handles judgment and review.

Chapter 02

Constraints & Decisions

Every architectural choice is a trade-off. These eight decisions define the shape of the system — and the boundaries it can never cross.

Constraint Profile

The system was built under hard constraints: one user, local machine only, career data that must never leave the device, free-tier LLM APIs, and no background in job queuing or stream processing.

ConstraintImpact on Design
Single user, local-onlySQLite instead of PostgreSQL — zero ops overhead, synchronous API fits Express handlers cleanly
Career data privacyNo auth, no sessions, no cloud hosting — localhost only, data never leaves disk except explicit API calls
Free-tier LLM costTwo-stage scoring (keyword gate first), JD truncation at 1500 chars, summary-based evaluation
No queue system knowledgeSequential on('close') chaining in scout.ts — simpler but no parallelism or retry isolation per stage
Python for LLM/PDF workspawn() child processes instead of inline Node.js — all Python scripts run as detached children
The Eight Key Decisions
Decision #1
Local-First Architecture
Cloud SaaS was rejected because career data is sensitive intelligence. Electron was rejected for bundle weight. The chosen path: Express + React served locally — zero packaging complexity, full filesystem access, privacy guaranteed. The target user is a technical job seeker who can clone a repo.
Trade-off accepted: Non-technical users cannot use this. No mobile. No multi-device access.
Decision #2
SQLite via better-sqlite3
PostgreSQL requires a running service and connection pooling — overkill for one user. better-sqlite3's synchronous API was the tipping point: Express route handlers that do db.prepare(...).run() are cleaner than async/await chains. The single-user constraint meant write contention was never a concern.
Trade-off accepted: No concurrent write safety, file-level locking, not portable to multi-user.
Decision #3
Python Pipeline as Spawned Child Process
All LLM orchestration, PDF compilation, and document drafting runs as Python — not because Node.js couldn't do it, but because all the code already existed in Python. Stdout capture with [LOG]/[FOUND]/[DONE] prefix conventions gives real-time log streaming to the UI without WebSockets.
Trade-off accepted: Stdout parsing is fragile. No mid-run cancellation. Exit code is the only reliable error signal.
Decision #4 — ADR-005
Materialized JSON for Pipeline Config
Python can't easily read SQLite without a connection. The solution: the Express server writes data/candidate_preferences.json whenever settings change. Python reads this file. SQLite is the source of truth; the JSON is a projection. The write lives in materializeJobSearchPrefs() in server/shared.ts.
Trade-off accepted: Two sources of truth. Drift risk if server crashes mid-write.
Decision #5 — CR-006
LLM Provider Fallback Chain
A single-provider design meant any rate limit or outage silently killed the pipeline. The fix: call_llm() checks which providers are configured, orders them primary-first, and iterates. Non-rate-limit errors skip to the next provider. Rate limits retry within the same provider with escalating waits.
Trade-off accepted: A job might be scored by Gemini and a second by Claude in the same batch run, producing inconsistent scoring.
Decision #6
Proof Code Anti-Hallucination System
Telling an LLM "only use facts from this document" is reliably ignored under token pressure. The structural fix: every claim in workExperience.md gets a stable code (ACC-101, VOC-01, MET-03). Drafting prompts require every claim to cite a code. If no code exists, the claim cannot be made. Implemented in codifyExperienceAndAssignIDs() in server/routes/profile.ts.
Trade-off accepted: User must invest time codifying their experience before the pipeline produces useful output.
Decision #7
Two-Stage Scoring (Zero-Token Gate + LLM)
A 50-job batch with LLM scoring on every job would cost 50 API calls. In practice, ~70% of jobs are obvious rejects (wrong title, wrong seniority). A deterministic keyword gate runs first — zero tokens, zero cost — and eliminates the noise. The LLM only evaluates jobs that pass the gate. Implemented in passes_jd_keyword_gate() in batch_pipeline.py.
Trade-off accepted: False negative risk at the gate layer. A role with an unusual title may be wrongly rejected.
Decision #8 — May 2026
Modular Express Router Architecture
server/index.ts had grown to 844 lines containing all 24 routes, four helper functions, and business logic. A duplicate copy of the API key-reading logic existed in scout.ts. The refactor split all routes by domain into server/routes/ (system, jobs, profile, pipeline) and extracted shared utilities into server/shared.ts. index.ts is now 33 lines — middleware setup and router mounts only. The refactor also fixed a latent bug: /api/jobs/:id/draft was missing buildPythonEnv(), so manual draft triggers never had API keys injected.
Trade-off accepted: Five files to open instead of one when debugging an unfamiliar route. Mitigated by naming that makes the domain obvious.

Data Schema

Three core tables. jobs is the canonical record of every discovered opportunity. profiles is a key-value store for all user configuration. activity_log is the append-only event log that drives the terminal view in the UI.

FieldTypePurpose

The Proof Code System

Every claim in workExperience.md gets a stable code: ACC-101 for accomplishments, VOC-01 for vocabulary/tools, MET-03 for metrics. Section 5.N maps to ID range N×100+1 through (N+1)×100 — so multiple employer sections never produce collisions.

The codifyExperienceAndAssignIDs() function lives in server/routes/profile.ts (moved from server/index.ts during the May 2026 router refactor). It scans existing IDs on every save to seed counters above current maxima — preventing collisions when a user resaves an already-codified document.

Why This Works

Structural constraints are more reliable than prompt constraints. Telling an LLM "only use facts from this document" is routinely ignored under token pressure. Requiring a valid code in every claim makes hallucination structurally impossible — if no code exists for a claim, the claim cannot appear in the output.

WYSIWYG Document Editor

Users open a PDF asset from the job detail panel. A full-screen overlay shows the PDF preview on the left and a rich-text Markdown editor on the right. On save, the pipeline runs style_compliance_guard.py then compile_single.py (Playwright headless Chromium) to regenerate the PDF. Route handler lives in server/routes/jobs.ts. An AI rewrite field routes through scripts/ai_rewrite.py.

The known failure mode on Windows: Playwright can't find the Chromium executable. Fix: npx playwright install chromium (Node) and playwright install chromium (Python) — both runtimes need separate installs.

Chapter 04

The LLM Layer

Five distinct use cases. One fallback chain. A total monthly cost under $0.15 — usually $0 on Gemini Flash's free tier.

Where LLMs Are Used
01
Job Fit Scoring

evaluate_job_fit() in batch_pipeline.py. Inputs: candidate preferences + workExperience_summary.md + truncated JD (1500 chars) + job_fit_engine.md rules. Outputs JSON: Decision, Score, Confidence, Summary, TopFitReasons, RiskFlags. response_mime_type="application/json" forces JSON output on Gemini.

~3,000–4,000 input tokens · ~$0.0003/call · 15 calls/batch ≈ $0.005
02
Asset Drafting

drafting_engine.py. Inputs: full workExperience.md + JD + company research + resume template + cover letter reference + style guides + claim verifier rules. Outputs: tailored resume Markdown + cover letter Markdown. Every claim must cite a proof code — no code, no claim.

~8,000–15,000 input tokens · ~$0.002/job · 5 qualified/batch ≈ $0.01
03
Company Research

research-engine.py. Uses Perplexity sonar-pro if configured (web-grounded), else falls back to Gemini with tools=[{"google_search": {}}] for grounding. Outputs a structured Research Packet: mission, financials, competitors, culture, recent news. When falling back to Claude, the tools parameter is intentionally dropped — quality degrades but the pipeline continues.

Per qualified job · Perplexity preferred for web grounding
04
Experience Summary Generation

generate_experience_summary.py. Runs as a detached background process after every experience save. Inputs the full workExperience.md and outputs a condensed workExperience_summary.md (~400–600 words). This summary is used for LLM scoring instead of the full file — reducing scoring token cost by ~80% for rejected jobs.

Background, on-save · ~$0.001 per generation
05
AI Document Rewrite

ai_rewrite.py. On-demand, user-triggered from the WYSIWYG editor. Accepts a natural language instruction + current document text, returns a rewritten version. Route handler in server/routes/jobs.ts writes instruction and text to scratch/ tmp files, spawns the script, captures stdout as the rewritten content.

On-demand · ~$0.001–0.003 per rewrite
Provider Fallback Chain

call_llm() checks which providers have valid API keys configured, orders them with the user's primary choice first, then iterates. A None return means "this provider failed non-fatally, try next." A string — even empty — means "this provider responded, use this result."

LLM Provider Fallback Chain — _get_configured_providers() in scripts/utils.py
Prompt Engineering Decisions

The scoring prompt injects the job_fit_engine.md rules file verbatim rather than summarizing it. This was an intentional choice — the rules are nuanced and any compression risks losing important scoring criteria.

What Failed First

An earlier prompt that asked the LLM to "evaluate fit holistically" produced inconsistent score distributions. Adding the explicit job_fit_engine.md rules file and specifying the exact JSON schema improved consistency dramatically. The key shift: replace natural language instructions with structural constraints wherever possible.

A notable abstraction gap: the job_fit_engine.md rules file governs scoring behavior but has no UI. Changes to the user's blocklist in Settings update candidate_preferences.json — but the LLM scoring rules remain static. They're "configuration" that is really a prompt component, and should be versioned and tested with the same rigor as actual code.

Chapter 05

Lessons & Debt

What was intentional, what was accidental, what calcified into load-bearing infrastructure, and what needs to change first.

Accidental vs. Intentional Decisions
Resolved — May 2026 844-Line God File (server/index.ts)

Started as a small Express server and grew feature by feature with no controller extraction. All 24 routes, codifyExperienceAndAssignIDs(), materializeJobSearchPrefs(), buildPythonEnv(), and more lived in one file. scout.ts had a duplicate copy of the key-reading logic. Hard to navigate; no separation between HTTP transport and business logic.

Resolution: Extracted to server/routes/ (system, jobs, profile, pipeline) + server/shared.ts for path constants and shared utilities. index.ts is now 33 lines — middleware setup and router mounts only. The refactor also fixed a latent bug: /api/jobs/:id/draft was missing buildPythonEnv(), so manual draft triggers never had API keys injected.

Active Callback Pyramid — Pipeline Chaining

server/scout.ts chains four process spawns via nested on('close') callbacks, four levels deep (lines 71–157). This is semantically correct but creates code that is impossible to add retry logic to, and very hard to add error recovery to individual stages.

Fix path: A simple SQLite-backed job queue, or convert to an async generator pattern. The boundary is already clear — execution is the blocker.

Active Filesystem as Message Queue (jobs/ directory)

scout_local.ts writes JD text to jobs/{company}.txt. batch_pipeline.py reads via glob.glob. Files are never cleaned up — stale text from old scout runs persists indefinitely and may be re-processed. No ordering guarantee. No atomic writes.

Fix path: Clean up processed files after each pipeline run, or store JD text directly in SQLite (an additional TEXT column on the jobs table).

Active No Schema Migrations

Schema is defined as CREATE TABLE IF NOT EXISTS in server/db.ts — idempotent on boot, but doesn't handle column additions. Every schema change requires a manual ALTER TABLE or dropping and rebuilding the database. Acceptable for a solo developer who is also the user; a hard blocker for any public distribution.

Fix path: Add a schema_version table and run numbered migration scripts on boot. Trivial to add now, painful after a user base exists.

Active Stdout Prefix Parsing as IPC Protocol

Python scripts communicate progress via print() with prefix conventions: [LOG], [FOUND], [REJECT], [ACTION], [DONE]. server/scout.ts parses these line-by-line. It works but is fragile — any stray print() that doesn't follow the convention is silently misinterpreted.

Fix path: Replace with newline-delimited JSON from Python. One JSON object per line, server JSON.parse()s each. Type-safe, structured, eliminates the prefix convention entirely.

Refactoring Roadmap
Phase 1 — Critical for Robustness
Schema migrations · jobs/ cleanup · activity_log LIMIT · JSON streaming
Add a schema_version table and run numbered migration scripts on boot. Clean up processed job text files after each pipeline run. Add LIMIT to activity_log queries and a periodic DELETE for old entries. Replace stdout prefix parsing with newline-delimited JSON from Python.
Phase 2 — Important for Reliability
Lightweight job queue · per-job LLM timeout · structured LLM logging
Introduce a SQLite-backed queue for pipeline stages — even a simple one eliminates the callback pyramid and enables per-stage retry. Add a per-job LLM call timeout to prevent a single slow call from blocking the entire batch. Log token counts and provider used per LLM call.

job_fit_engine.md UI · Playwright browser reuse
Build a UI for editing the scoring rules file — it's currently a prompt artifact with no UI surface, invisible to any user who isn't also the developer. Implement Playwright browser pool/reuse across scout stages to reduce cold start overhead.

Key Learnings

What a multi-month build teaches that a weekend project cannot.

Patterns Worth Keeping
Materialized View Pattern

Write to SQLite (source of truth), then project to a JSON file that downstream consumers can read without DB knowledge. SQLite owns the write path; Python owns the read path. Implemented in materializeJobSearchPrefs() in server/shared.ts. This pattern is correct for any multi-runtime architecture where one runtime needs config from another.

Structural Anti-Hallucination

Proof codes work because they make hallucination structurally impossible, not just instruction-bound. The general principle: when you need an LLM to respect a constraint reliably, encode the constraint in the data structure rather than the prompt. "Only use facts from this document" is ignored. "Every claim must cite a code that exists in this document" is enforceable.

Two-Stage Evaluation

Zero-cost gate first, expensive gate second. In a 50-job batch, ~70% of jobs are eliminated by a deterministic keyword check before any LLM call. This pattern applies to any pipeline with a large input set and expensive evaluation: pre-filter with a cheap heuristic, then apply the expensive model only to candidates that pass.

Domain-Grouped Router Split

When an Express server grows beyond ~200 lines, splitting by domain (system, jobs, profile, pipeline) with a shared utilities module is more useful than splitting by layer (controllers, services, routes). The domain split reflects how you actually debug and navigate the code; a layer split just moves the sprawl into more files.

Mistakes & Lessons
Build the IPC protocol first

The [LOG]/[FOUND] stdout prefix convention was a hack that became load-bearing. Defining a JSON streaming protocol from day one would have cost one hour and saved many debugging sessions. Anything that touches a process boundary deserves a schema.

Add migrations before the first schema change

SQLite ALTER TABLE is limited. Columns added after the fact require manual migrations or data loss. A migration table costs almost nothing to add at the start; it costs real effort to retrofit around a growing database.

Extract route handlers immediately

server/index.ts grew to 844 lines because each feature was "just one more route." The domain boundaries were always obvious — it just took the pain of a 844-line file to enforce them. The May 2026 refactor also uncovered a latent bug that had been invisible in the monolith.

Prompt engineering is code

job_fit_engine.md is a prompt component masquerading as a config file. Changes to it immediately change scoring behavior with no code change — which means it should be versioned, tested, and treated with the same rigor as actual code.

What Would Be Built Differently

Start with a lightweight job queue (even a simple SQLite-backed one) for pipeline stages. Define a structured JSON streaming protocol for Python → Node communication from day one. Extract route handlers into domain modules immediately. Add a migration system before writing the first schema. Keep the proof code system, the two-stage scoring, the materialized JSON projection, SQLite, and the domain-grouped router pattern — those are all right for this architecture.
